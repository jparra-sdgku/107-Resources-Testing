A
Arrange Pattern
The first step of the AAA pattern. Set up everything the test needs before running it — create objects, set the initial state, and prepare the inputs. Think of it as the "before" state of the test.
val viewModel = TodoViewModel() // Arrange: create a fresh ViewModel
Related: AAA Pattern, Act, Assert View Flashcard →
Act Pattern
The second step of the AAA pattern. Call the function or perform the action you want to test. Each test should have only one Act — one single thing being verified.
viewModel.addTask("Buy Groceries") // Act: call the function under test
Related: AAA Pattern, Arrange, Assert View Flashcard →
AAA Pattern Pattern
Arrange, Act, Assert — the three-step structure that every well-written test follows. It makes tests easy to read, understand, and maintain. The pattern separates setup, execution, and verification into clear phases.
val vm = TodoViewModel() // Arrange vm.addTask("Buy Groceries") // Act assertEquals(1, vm.getTaskCount()) // Assert
Related: Arrange, Act, Assert View Flashcard →
Assert Pattern
The third step of the AAA pattern. Check that the result is what you expected using assertions. If the assertion fails, the test fails and reports exactly what went wrong.
assertEquals(1, viewModel.getTaskCount()) // Assert: check the result
Related: AAA Pattern, Assertion View Flashcard →
Assertion Assertion
A check inside a test that verifies a specific condition. It says "I expect this value to be X." If the condition is not met, the test fails immediately and reports the expected vs actual value. JUnit provides many assertion functions like assertEquals, assertTrue, assertFalse, and assertNotNull.
assertEquals(5, result) // fails if result is not 5 assertTrue(list.isNotEmpty()) // fails if list is empty
Related: assertEquals, assertTrue View Flashcard →
assertEquals Assertion
A JUnit assertion that checks whether two values are equal. Takes two parameters: the expected value first, then the actual value. If they are not equal, the test fails with a message showing what was expected and what was actually received. Order matters — always put the expected value first.
assertEquals(1, viewModel.getTaskCount()) // expected: 1 — actual: viewModel.getTaskCount() // fails with: "expected: <1> but was: <0>"
Related: Assertion, assertTrue View Flashcard →
assertTrue Assertion
A JUnit assertion that checks whether a condition is true. Takes one parameter — any expression that returns a Boolean. If the expression returns false, the test fails. More readable than assertEquals(true, ...) for boolean checks.
assertTrue(viewModel.containsTask("Buy Groceries")) // passes if containsTask() returns true // fails if containsTask() returns false
Related: Assertion, assertEquals View Flashcard →
Automated Testing Concept
Tests written in code that run automatically without human intervention. Automated tests are faster, more consistent, and can run on every code change 24 hours a day. They catch regressions immediately and cover edge cases that humans often forget to test manually.
// Runs automatically when you push code: ./gradlew test // runs all unit tests in seconds
Related: Manual Testing, Regression View Flashcard →
B
Bug Life Cycle Concept
The predictable journey every bug takes from discovery to resolution: Found → Reported → Assigned → Fixed → Verified → Closed. Understanding this cycle is fundamental for anyone working in QA or software development. The developer often writes an automated test when fixing a bug to prevent it from coming back.
// Bug life cycle in practice: // 1. Found: "App crashes when adding a blank task" // 2. Reported: screenshot + steps to reproduce // 3. Assigned: developer investigates // 4. Fixed: add isNotBlank() check + write a test for it // 5. Verified: QA confirms fix works // 6. Closed: removed from backlog
Related: Regression, QA Engineer View Flashcard →
C
Code Coverage Concept
The percentage of your code that is executed by your tests. 80% coverage means 80% of your code is tested by at least one test. Higher coverage generally means fewer untested paths where bugs can hide. Coverage is a useful metric but not the only indicator of test quality — 100% coverage does not mean the tests are well written.
// If addTask() has 3 lines and your tests execute all 3: // Coverage for addTask() = 100% // If only 2 lines are executed: Coverage = 66%
Related: Unit Test, Test Case View Flashcard →
Compose Testing Tool
The testing library for Jetpack Compose UI tests. It works alongside Espresso and provides a modern API for interacting with composable functions. Uses composeTestRule to launch composables in a test environment, and functions like onNodeWithTag, onNodeWithText, performClick, and assertIsDisplayed to simulate user interactions and verify what appears on screen.
composeTestRule .onNodeWithTag("addButton") .performClick() composeTestRule .onNodeWithText("Buy Groceries") .assertIsDisplayed()
Related: Espresso, testTag, UI Test View Flashcard →
D
Developer Testing Role
Testing done by the developer who wrote the code. Focuses on logic correctness using unit and integration tests. Tests run automatically as part of the development workflow. The goal is to verify that the code does exactly what it is supposed to do — not to find bugs from a user's perspective.
// Developer writes tests for their own functions: @Test fun addTask_withValidTitle_appearsInList() @Test fun addTask_withBlankTitle_isIgnored() @Test fun removeTask_removesCorrectItem()
Related: QA Engineer, Unit Test View Flashcard →
E
Espresso Tool
Google's official framework for writing UI tests in Android. It simulates real user actions — tapping, typing, scrolling — and verifies what appears on screen. Since this course uses Jetpack Compose, we use the Compose Testing library which is built on top of Espresso. UI tests using Espresso require a running emulator or device.
// Espresso / Compose Testing in androidTest/: composeTestRule.onNodeWithTag("inputField") .performTextInput("Buy Groceries") composeTestRule.onNodeWithTag("addButton") .performClick()
Related: Compose Testing, UI Test, JUnit 4 View Flashcard →
I
Integration Test Concept
A test that checks multiple components working together correctly — not each piece in isolation, but the connections between pieces. For example, verifying that the ViewModel correctly saves a task to the database AND retrieves it back. Slower than unit tests and often requires some Android APIs. Lives in src/androidTest/java/.
// Integration test — tests ViewModel + Repository + Database together: @Test fun addTask_savesToDatabase_andRetrievesCorrectly() { viewModel.addTask("Buy Groceries") val fromDb = repository.getAll() assertEquals("Buy Groceries", fromDb[0].title) }
Related: Unit Test, UI Test, Testing Pyramid View Flashcard →
J
JUnit 4 Tool
The standard testing framework for Java and Kotlin. In Android, it is used to write unit tests for classes and functions that do not depend on Android-specific components. JUnit finds all functions marked with @Test and runs them automatically, reporting results with green checkmarks (pass) or red X marks (fail). Runs on the JVM — no device or emulator needed.
// JUnit test in src/test/java/: @Test fun addition_isCorrect() { assertEquals(4, 2 + 2) // passes instantly, no device needed }
Related: @Test, Assertion, Unit Test View Flashcard →
M
Manual Testing Concept
Testing done by a human who opens the app and clicks through it manually. It feels natural but has serious limitations: it is slow, inconsistent, misses regressions, and cannot run automatically. Manual testing is best for exploratory testing and UX evaluation. It cannot replace automated testing for ensuring code quality over time.
// Manual testing: developer opens app, types "Buy Groceries", taps Add // Problem: developer might forget to test the blank title case // Solution: automated test always covers every case, every time
Related: Automated Testing, Regression View Flashcard →
Mock Concept
A fake object that replaces a real one during testing. Mocks allow you to test your code in isolation without depending on external systems like databases, networks, or APIs. For example, a mock database always returns the same fixed data so you can test your logic without connecting to a real database.
// Real database: slow, requires setup, unpredictable // Mock database: instant, always returns what you tell it to val mockRepository = MockTodoRepository() mockRepository.returnTasks(listOf(Task(1, "Buy Groceries")))
Related: Unit Test, Integration Test View Flashcard →
Q
QA Engineer Role
Quality Assurance Engineer — a professional who tests the whole product from the user's perspective. Unlike developer testing which focuses on code correctness, QA engineers find edge cases that developers missed and verify the full user experience. They write test cases, report bugs with detailed steps to reproduce, and verify fixes. A well-paid and valuable career path in software development.
// QA Engineer finds: // "The app crashes when you paste text instead of typing" // — an edge case the developer never thought to test manually
Related: Developer Testing, Bug Life Cycle View Flashcard →
R
Regression Concept
When a new code change breaks something that used to work correctly. Regressions are one of the most common and frustrating bugs in software development — they appear in code that was already working. Automated tests catch regressions the moment they happen, because the tests for the previously working feature will now fail.
// addTask() worked fine last week // Developer changes Task.kt to add a new field // Now addTask() silently fails — a regression // The unit test for addTask() fails immediately → regression caught
Related: Automated Testing, Unit Test View Flashcard →
T
@Test Annotation
A JUnit annotation that marks a Kotlin function as a test. JUnit scans the test class for all functions marked with @Test and runs them automatically. Without @Test, the function is just a regular Kotlin function that JUnit will never find or run. Always required on every test function.
@Test // ← JUnit sees this and runs the function automatically fun addTask_withValidTitle_appearsInList() { // test code here } fun helperFunction() { // ← No @Test — JUnit ignores this // utility code }
Related: JUnit 4, @Before View Flashcard →
@Before Annotation
A JUnit annotation that marks a function to run automatically before every single @Test function in the class. Used to set up the initial state that all tests need — like creating a fresh instance of the class being tested. Runs once per test, not once per class, ensuring each test starts from the same clean state.
@Before fun setUp() { viewModel = TodoViewModel() // fresh ViewModel before EVERY test } // This runs before addTask_test(), before removeTask_test(), before ALL tests
Related: @Test, JUnit 4 View Flashcard →
Test Case Concept
One specific scenario being tested. A single function can have multiple test cases — one for the normal case and one or more for edge cases. For example, addTask() has test cases for a valid title, a blank title, a title with spaces, and an empty string. Each test case is a separate @Test function.
// Test cases for addTask(): @Test fun addTask_withValidTitle_appearsInList() // normal case @Test fun addTask_withBlankTitle_isIgnored() // edge case @Test fun addTask_titleIsTrimmed() // edge case @Test fun addTask_withEmptyString_isIgnored() // edge case
Related: Unit Test, @Test View Flashcard →
Testing Pyramid Concept
The industry-standard model for how many tests to write at each level. Wide at the bottom (many unit tests), narrower in the middle (fewer integration tests), narrow at the top (few UI tests). The shape communicates that unit tests are fastest and cheapest, while UI tests are slowest and most expensive to maintain.
// ▲ UI Tests (few — slow, need emulator) // ▲▲▲ Integration Tests (some — medium speed) // ▲▲▲▲▲▲▲ Unit Tests (many — fast, no device)
Related: Unit Test, Integration Test, UI Test View Flashcard →
testTag Tool
A Jetpack Compose modifier that assigns a name to a composable element so that UI tests can find it reliably. Without testTag, UI tests have no guaranteed way to locate a specific element on screen. Added with .testTag("name") in the app code and accessed with onNodeWithTag("name") in tests. Names are case-sensitive.
// In app code (TodoScreen.kt): OutlinedTextField( modifier = Modifier.testTag("inputField") // give it a name ) // In test code (TodoScreenTest.kt): composeTestRule.onNodeWithTag("inputField") // find it by name .performTextInput("Buy Groceries")
Related: Compose Testing, UI Test View Flashcard →
U
Unit Test Concept
A test that verifies a single function or class in complete isolation — no database, no network, no UI. It is the fastest and most numerous type of test in the Testing Pyramid. Unit tests run on the JVM (no device needed) in milliseconds. They are best for testing ViewModel logic, data transformations, and any function with clear inputs and outputs. Lives in src/test/java/.
// Unit test — no emulator, runs in milliseconds: @Test fun addTask_withValidTitle_appearsInList() { val viewModel = TodoViewModel() // no Android framework needed viewModel.addTask("Buy Groceries") assertEquals(1, viewModel.getTaskCount()) }
Related: JUnit 4, Integration Test, Testing Pyramid View Flashcard →
UI Test Concept
A test that simulates a real user interacting with the app — typing, tapping buttons, scrolling — and verifies what appears on screen after each action. The slowest type of test in the pyramid because it requires a running emulator or device. Best for testing complete user flows and verifying that UI elements respond correctly. Lives in src/androidTest/java/.
// UI test — needs a running emulator: @Test fun addTask_appearsInList() { composeTestRule.onNodeWithTag("inputField") .performTextInput("Buy Groceries") composeTestRule.onNodeWithTag("addButton").performClick() composeTestRule.onNodeWithText("Buy Groceries").assertIsDisplayed() }
Related: Espresso, Compose Testing, testTag View Flashcard →
